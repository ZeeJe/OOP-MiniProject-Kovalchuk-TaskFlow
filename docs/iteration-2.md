# Звіт по Ітерації 2 (Lab 35)

## 1. Готові Use Cases
- Створення задачі з валідацією.
- Зміна статусу задачі (з дотриманням бізнес-правил, напр., заборона `Done` -> `ToDo`).
- Асинхронне збереження та відновлення стану застосунку за допомогою JSON.
- Аналітичні вибірки (LINQ: фільтрація невиконаних, групування статистики, сортування).

## 2. Зміни в контрактах та класах
- Інтерфейс `ITaskRepository` розширено методами `Update()`, `SaveToFileAsync()`, `LoadFromFileAsync()`.
- Створено `JsonTaskRepository`, який використовує `Dictionary<Guid, TaskItem>` для швидкого доступу O(1).
- До класу `TaskItem` додано атрибути `[JsonInclude]` та `[JsonConstructor]` для коректної десеріалізації закритих властивостей.

## 3. Патерн розширення
Було застосовано патерн **Strategy** (Стратегія). 
- **Чому:** Для розрахунку "Терміновості" (`CalculateUrgency`). 
- **Що дає:** Дозволяє в майбутньому (на Lab 37) легко змінити формулу розрахунку пріоритетів (наприклад, враховувати дедлайни), створивши нову стратегію, при цьому не змінюючи ні доменну сутність `TaskItem`, ні сам `TaskService`.

## 4. Ризики для тестування (Підготовка до Lab 36)
- Найбільший ризик на наступній ітерації становлять інтеграційні тести файлової системи (`JsonTaskRepository`), оскільки реальний файловий I/O може кидати специфічні винятки (немає прав доступу, файл зайнятий іншим процесом).
- У Lab 36 необхідно покрити ці крайові сценарії I/O та налаштувати більш жорсткий Quality Gate (Coverage).
